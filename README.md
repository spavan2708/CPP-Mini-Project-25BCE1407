## рҹҸҰ Overview

**Imperial Bank** is a DA2 mini-project for the *Object-Oriented Programming* course at **VIT Chennai**. It simulates a real-world cooperative savings account system with full CRUD operations, persistent file storage, and a premium terminal-style web UI вҖ” all powered by a C++ backend compiled to **WebAssembly** via Emscripten.

The system demonstrates core C++ OOP concepts: **encapsulation**, **data hiding**, **abstraction**, **modularity**, and **STL usage** вҖ” while being fully accessible from any browser with no installation required.

### Class Breakdown

| Class | Responsibility | OOP Concept |
|-------|---------------|-------------|
| `Transaction` | Immutable record of one financial event | Data encapsulation |
| `Account` | Private balance вҖ” modified only via `deposit()`/`withdraw()` | Encapsulation + Data hiding |
| `BankSystem` | Orchestrates all operations, file persistence, STL containers | Abstraction + Modularity |

### Why C++ over C?

| C | C++ (This Project) |
|---|---------------------|
| `struct` with no methods | `class` with private data + public methods |
| Manual memory management | `vector<T>` вҖ” automatic, safe, resizable |
| Global functions for logic | Member functions вҖ” data and behavior together |
| No constructor guarantees | Constructors ensure valid initial state |
| No encapsulation | `private:` blocks prevent direct balance tampering |

## рҹ“җ STL Usage

| Container | Used For | Why Not Array? |
|-----------|----------|----------------|
| `vector<Account>` | Dynamic list of all accounts | No fixed size; supports `emplace_back`, `find_if`, `erase` |
| `vector<Transaction>` | Transaction log | Grows unboundedly; easy filter/slice for history |
| `string` | All text fields | Handles arbitrary length, `.trim()`, concatenation safely |
| `stringstream` | CSV parsing for file I/O | Clean tokenizing without `strtok` memory risks |
| `ostringstream` | Formatting TXN ref codes | Type-safe `setw`/`setfill` formatting |

## рҹӣ пёҸ Challenges Faced

**1 В· Emscripten stdin blocking** вҖ” `cin >>` occupies the WASM thread and blocks the JS event loop entirely. Fixed by replacing all `cin >>` with `getline()`-based helpers and adding `WASM_YIELD()` macros with `-sASYNCIFY=1`.

**2 В· `cin.ignore()` + `getline()` bug** вҖ” After `cin >> accNo`, a `\n` is left in the buffer causing `getline` to return empty on the next read. Resolved by eliminating all `cin >>` and using only line-based reads.

**3 В· OOP design challenge** вҖ” Initially put file I/O logic inside `Account`. Refactored to keep `Account` purely about data/behavior, with `BankSystem` owning all persistence вҖ” proper separation of concerns.

**4 В· GitHub Pages 404** вҖ” First deploy failed silently. Root cause: repo was private. Fixed by making it public and re-enabling Pages.

**5 В· WebAssembly CORS restrictions** вҖ” `file://` protocol blocks WASM loading. Required running a local Python server (`python -m http.server`) during development.

**6 В· Balancing WASM + JS** вҖ” The web frontend (JS) and the C++ backend (WASM) needed to stay in sync. Solved by keeping C++ as the source of truth for business logic and mirroring the class design in JS for the live UI layer.
